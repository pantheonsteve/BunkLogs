# Internationalization Layers

Three distinct concerns that should not be conflated.

## The three layers

| Layer | What it is | Where it's specified | Tier 1 scope |
|---|---|---|---|
| **Content language** | The language the staff member writes in | `Reflection.language`, `Note.language` fields on the data model | English, Spanish, Hebrew (content) |
| **UI translation** | App labels, buttons, errors in the user's language | Frontend i18n strings via `react-i18next` | English, Spanish (Kitchen Staff surfaces in Tier 1) |
| **Auto-translation** | LLM-generated English of non-English content for leadership readers | Server-side Celery task using Anthropic API | English-only target for Tier 1 |

## Content language

Captured per submission as a metadata field. The platform does not constrain which language a user writes in; the field records what they used. Readers see the original in the original language plus the auto-translation if applicable.

## UI translation

`react-i18next` with translation files in the repo. Tier 1 UI translation scope:

- Kitchen Staff dashboard (Story 37) — English, Spanish
- Language selector (Story 38) — English, Spanish; Hebrew label rendered natively (עברית) even though full UI is not translated
- Reflection form chrome (Save, Submit, validation, audience disclosure)
- Reflection history view
- Sign-in error states for misconfigured authenticated users

Other surfaces (UH dashboards, admin surfaces) remain English-only for Tier 1. Tier 2 expands UI translation as non-English-preferred users reach those surfaces.

## Hebrew and RTL scope

Hebrew is supported for **content** in Tier 1 — Israeli kitchen staff can write reflections in Hebrew, with proper bidi rendering at the text level. Hebrew **UI** is deferred to Tier 2 because full right-to-left layout support across React + Tailwind + MUI is a substantial engineering investment incompatible with the June 5 deadline. A Hebrew speaker uses the English UI to write Hebrew content in Tier 1.

## Auto-translation

When a Reflection or Note is submitted with `language ≠ 'en'`, a Celery task generates an English translation server-side using the Anthropic API. The translation is stored alongside the original on the record, with metadata: source language, target language (English), model ID, timestamp.

**Translation prompt** (used for both reflections and notes):

```
Translate the following text from {source_language} to English. Preserve meaning faithfully; do not embellish or summarize. Translate culturally specific terms with the original in brackets when no clean English equivalent exists. Preserve the structure of the input (paragraphs, lists, line breaks). Return only the translation, no explanatory wrapper.

{content}
```

**Retry policy:** exponential backoff with 3 attempts (1 min, 5 min, 30 min). After 3 failures, marked permanently failed; readers see "Translation unavailable — retry" state.

**Re-translation on edit:** cancels any pending task, enqueues fresh task. Prior translations retained per audit policy.

**Cost monitoring:** Datadog metrics `bunklogs.translation.submitted`, `bunklogs.translation.completed`, `bunklogs.translation.failed`, `bunklogs.translation.tokens_used`.

**Audit retention:** 90 days of prior translations retained per Reflection/Note, then garbage-collected.

## Reader-side display

Per Story 44 (Kitchen Staff) and Story 47 (LT), readers of non-English content see:

- **Translated** state — English translation primary, original collapsible
- **Pending** state — original primary, "Translating to English — refresh in a moment"
- **Failed (retryable)** — original primary, "Translation unavailable — retry"
- **Failed (terminal)** — original primary, "Translation could not be generated. Contact Admin"

A per-reader preference ("Translation first" vs. "Original first") persists across sessions for bilingual readers.

## What's not in scope for Tier 1

- Cross-language pivots (Spanish-to-Hebrew, etc.)
- Translation confidence/quality flagging
- Live draft translation as user types
- Translation request workflow (LT requests translation from native speaker)
- Multilingual UI for any role surface other than Kitchen Staff
- Full RTL layout support
