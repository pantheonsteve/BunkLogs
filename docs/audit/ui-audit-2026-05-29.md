# BunkLogs UI Audit — 2026-05-29

## Executive Summary

The frontend is in a **strangler-fig migration state**: new multi-tenant role flows under `/counselor`, `/unit-head`, `/camper-care`, and related paths coexist with legacy Crane Lake surfaces (`/bunk/*`, `/counselor-dashboard`, `/unithead/*`, `/admin-bunk-logs`). The **Counselor and Unit Head daily loops are largely implemented** on the new routes and are shippable for happy-path Summer 2026 use, but several story-level gaps remain (offline tolerance, request detail views, draft autosave, read-only-then-edit for camper reflections).

The biggest architectural risks are **dual implementations of the same concept** (two bunk stacks, two camper drill-down paths), **navigation gated on legacy `User.role` strings** while capability-based RBAC is only partially wired, and **inconsistent layout/error patterns** across AppLayout routes vs. standalone pages. Notes platform UI is present (`/notes`, `/subject-notes`) but cross-reference flows from bunk concerns and specialist notes are not fully wired.

Most important next steps for Wave 2 UX (FA-G/H/I): (1) retire or hard-redirect legacy routes, (2) pick one camper profile surface and wire bunk drill-down consistently, (3) add sidebar home links and post-login routing for Specialist/Maintenance/Kitchen/Madrich, (4) unify loading/error/empty-state patterns, (5) close blocking story gaps on counselor request detail and UH order/specialist drill-down.

### Role-based routing

Three mechanisms overlap:

1. **`ProtectedRoute`** — auth gate only; no role check.
2. **`AdminRoute`** — `isSuperAdmin(user) || user.role === 'admin'`.
3. **Capability gates in Sidebar** — `hasCapability(user, tier)` from legacy `User.role` mapping.
4. **Per-page role gates** — e.g. legacy `UnitHeadDashboard.jsx` checks `user.role === 'Unit Head'`.
5. **Post-login redirect** — `Dashboard.jsx` maps `User.role` → role home via `ROLE_DESTINATIONS`; Admin falls through to legacy `BunkGrid`.

`postLoginRedirect.js` (`safeNextPath`, OAuth `?next=` persistence) exists and is unit-tested but **is not imported by Signin or auth callbacks** — login always lands on `/dashboard` → role redirect.

### Auth context

- `useAuth()` exposes `user`, `isAuthenticated`, `loading`.
- User profile comes from JWT + `/api/v1/users/email/:email/` or stored `localStorage.user_profile`.
- **`Membership.capability` is not on the auth context yet** — `capability.js` maps legacy Title Case `User.role` strings.
- Known limitation: `health_center` and `special_diets` both surface as `'Camper Care'` and cannot get distinct nav.

### Sidebar IA (3.32)

| Section | Gate | Links |
|---------|------|-------|
| **My work** | All authenticated | Home, My tasks, role home (Counselor/UH/CC only), File a reflection, My reflections, Notes, Subject notes |

**Nav gaps:** No sidebar entries for Specialist, Maintenance, Kitchen Staff, Madrich, or Unit Head home (UH gets `/unit-head` only via Dashboard redirect, not sidebar — actually UH does get sidebar link at line 196-201). Specialist/Maintenance/Kitchen/Madrich must bookmark or rely on post-login redirect.

### Layout coverage

- **AppLayout:** `/reflect`, `/tasks`, `/notes`, counselor/UH/CC/specialist/kitchen/madrich/LT/maintenance flows, `/dashboards/group/:id`, subject-trends.
- **AdminLayout:** `/admin/*` children (template editor at `/admin/templates/:id/edit` is intentionally full-bleed outside layout).
- **Standalone chrome:** `/dashboard`, legacy bunk/camper pages, several `/dashboards/*` analytics pages (`SubjectDetailPage`, `DashboardsHub`, etc.) mount Sidebar+Header inline — inconsistent with AppLayout children.
- **No bottom nav** — mobile users get collapsible sidebar + header only.

## §5 — Design System and Consistency

### Component library

- **Tailwind CSS v4** is the primary styling system across all audited surfaces — utilities, dark mode, responsive breakpoints.
- **`@mui/material`** is listed in `package.json` but has **zero imports** in `frontend/src` — dead dependency.
- **Other UI:** Radix popover, Lucide icons (admin), Quill (rich text), react-day-picker, Chart.js (legacy only).

### Navigation chrome

- Most new role flows share **AppLayout** (sidebar + header).
- **Inconsistency:** `/dashboard`, legacy bunk/camper, several `/dashboards/*` pages, `/my-reflections`, `/reflections/:id` mount Sidebar+Header manually or without AppLayout.
- **Role-filtering:** Sidebar uses capability tiers + explicit `user.role` checks for Counselor/UH/CC home links. Specialist/Maintenance/Kitchen/Madrich users see generic "My work" without a role home shortcut.
- **No nav leakage observed** between Counselor and UH sections — gates appear correct for capability tiers.

### Mobile responsiveness

- **Strengths:** `min-h-[44px]` touch targets on new role pages; `pb-24` bottom padding; collapsible sidebar with overlay on mobile; ScoreGrid sticky name column + horizontal scroll.
- **Weaknesses:** AppLayout sidebar (not bottom nav); legacy dashboards desktop-oriented; ScoreGrid requires horizontal scroll with no card fallback; NoteComposer uses centered modal not bottom sheet; `ReflectionField` traffic-light buttons may overflow on ~320px screens; ThreadView uses `h-screen` under AppLayout causing layout jump.
- **No pull-to-refresh** — counselor/UH use 60s polling.

### Loading and error states

- **Fragmented:** Admin pages use `ErrorPanel`; role flows use inline amber/red divs or plain text.
- **`DashboardLoading` orphaned** — never imported.
- API errors generally surfaced as user-readable strings; some archive failures silent (ThreadView).

### Empty states

- Present on many dashboards (no bunks, no tasks, no notes).
- Madrich empty state for unassigned templates specified in Story 63 — verify at runtime (code suggests single-template card pattern).
- Classroom group type shows explicit stub empty state.

### Typography and spacing

- Generally consistent Tailwind scale on new pages (`px-4 sm:px-6 lg:px-8`, `max-w-3xl` on mobile-first flows).
- Legacy pages use wider `max-w-9xl` layouts — visually distinct generation.
